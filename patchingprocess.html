Patching Process in Solaris

Patching in Solaris is a critical procedure used to keep the operating system up-to-date with the latest security patches, bug fixes, and system improvements. The patching process involves using Oracle-provided patch bundles and applying them using commands like 'pkg' to update system packages. There are two main types of patches: Base Repo Patch and SRU (Support Repository Update) Patch.

1. Base Repo Patch

Base Repo Patch refers to the core patch repository, which contains the essential patches for Solaris OS. These patches are usually more general updates to the Solaris environment, such as security updates, kernel fixes, or enhancements.

2. SRU Patch

SRU patches are specific patches that are used for support and stability updates. These patches address particular system components and offer fixes or improvements. They are bundled into SRU files that can be downloaded from Oracle's My Oracle Support (MOS).

Before beginning the patching process, we need to perform several preparatory steps, such as checking the system state, identifying the patch versions, and ensuring the server is ready for patching.

Steps to patching process in Solaris:

Step 2: Downloading Patch Bundle

After completing the prechecks, the next step is to download the patch bundle for either Base Repo or SRU Repo from Oracle's My Oracle Support (MOS). You will need an active support contract with Oracle to download the patches.

Once the patches are downloaded, use FTP tools like WinSCP, FileZilla, or any other FTP/SFTP tool to copy the downloaded patch bundle to the Solaris server.

After transferring the patch bundle, extract the contents of the installation guide.

[root@localhost:~]# unzip p37322549_1100_SOLARIS64.zip

[root@localhost:~]#

The extracted installation guide contains the script file 'install-sru.ksh', which will uncompress the patch repository files.

Before running the script, create an empty directory to store the uncompressed Base or SRU repository data:

[root@localhost:~]# mkdir /opt/sru11.4.69

Once the directory is created, run the installation script and specify the path of the newly created directory to store the uncompressed data.

Step 3: Setting Up Publisher

Now that the repositories are uncompressed, we need to set up the publisher for the new repository. Publishers are responsible for identifying the patch repositories and providing the OS with the correct path.

To view the currently available publishers, use the following command:

[root@localhost:~]# pkg publisher
PUBLISHER TYPE STATUS P LOCATION
solaris origin online F file:///mnt/SRU11.4.69/
[root@localhost:~]#

Now, to remove any old publisher configurations, use the following command:

[root@localhost:~]# pkg unset-publisher solaris

Next, set the new publisher for both the SRU and Base repositories by providing the correct path to the repository. For example:

[root@localhost:~]# pkg set-publisher -g file:///opt/SRU11.4.76 solaris
[root@localhost:~]# pkg set-publisher -g file:///opt/Base11.4 solaris

Step 4: Dry Run for Patching

Before performing the actual patching, itâ€™s recommended to perform a dry run. A dry run simulates the patching process without making any changes to the system. It shows which packages will be updated, removed, or added, and provides an overview of the space required and any potential errors.

To perform a dry run, use the following command:

[root@localhost:~]# pkg update -nv

[root@localhost:~]#

The '-nv' flags stand for "no install" and "verbose," respectively. This allows you to preview the patching actions and confirm if everything looks correct.

Step 5: Final Patching Operation

After confirming that the dry run completed successfully and no errors are found, you can proceed with the actual patching operation. The final patching operation will update the repositories and create a new boot environment.

To perform the final patching operation, run:

[root@localhost:~]# pkg update --accept

The '--accept' flag accepts all patch dependencies and proceeds with the update.

Step 6: Boot Environment

After the patching operation, a new boot environment (BE) will be created. The boot environment allows you to revert to a previous configuration if the new patches cause any issues.

You can view the available boot environments using the command:

[root@localhost:~]# beadm list
BE Name Flags Mountpoint Space Policy Created
------------- ----- ---------- ------- ------ ----------------
11.4.69.170.2 N / 138.24G static 2024-03-17 15:37
solaris - - 1.48G static 2024-02-23 17:37
11.4.76.170.1 R / 138.24G static 2024-12-17 13:31
[root@localhost:~]#

The output will show all the available boot environments and their current status. The 'N' flag indicates the currently running BE, while the 'R' flag indicates the BE that will be activated after a reboot. A 'NR' flag means that the BE is both running and will be active after the next reboot.

If you want to activate a different boot environment, use:

[root@localhost:~]# beadm activate 11.4.69.170.2

Step 7: Rebooting the System

After completing the patching process and confirming the new boot environment is set, a reboot is required to apply the patches. Until the system is rebooted, the new patches wonâ€™t be activated.

To reboot the system, use the following command:

[root@localhost:~]# init 6

After the system reboots, it will come up with the new SRU according to the selected boot environment. You can verify the current boot environment again after the reboot:

[root@localhost:~]# beadm list
BE Name Flags Mountpoint Space Policy Created
------------- ----- ---------- ------- ------ ----------------
11.4.69.170.2 - / 138.24G static 2024-03-17 15:37
solaris - - 1.48G static 2024-02-23 17:37
11.4.76.170.1 NR / 138.24G static 2024-12-17 13:31
[root@localhost:~]#

You can also check the updated version of the OS by running:

[root@localhost:~]# uname -a

Roll Back to Previous Version

If you encounter issues with the new patch, or if you want to roll back to the previous version, you can revert to a previous boot environment (BE) using the following steps:

Condition 1: If Application Has Issues with New OS Patch

First, check the previous boot environment status using the command:

[root@localhost:~]# beadm list
BE Name Flags Mountpoint Space Policy Created
------------- ----- ---------- ------- ------ ----------------
11.4.69.170.2 N / 138.24G static 2024-03-17 15:37
solaris - - 1.48G static 2024-02-23 17:37
11.4.76.170.1 R / 138.24G static 2024-12-17 13:31
[root@localhost:~]#

Next, select the previous boot environment and activate it:

[root@localhost:~]# beadm activate 11.4.69.170.2

It will set the flag 'R' on the previous BE. After this, reboot the server to activate the previous BE:

[root@localhost:~]# init 6

Condition 2: If the System Is Not Booting or Not Coming Up

If the system is stuck and is not booting, you can access the system via console login. The system will show the 'ok >' prompt.

ok >

From here, list down all the available boot environments using the following command:

ok > boot -L

It will show all available boot environments. You can then choose and boot from the previous boot environment using the following command:

ok > boot 11.4.69.170.2

This will boot the system with the previously functional BE.
